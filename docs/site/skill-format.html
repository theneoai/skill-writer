SKILL.md Format

YAML-based skill specification format for AI skill lifecycle management. Each skill is defined by a structured YAML document with sections, triggers, and evaluation criteria.

📋
Universal Metadata
required
name*
Unique skill identifier (kebab-case)
version*
Semantic version (semver.org)
description*
Bilingual: {en: string, zh: string}
license
SPDX identifier (e.g., MIT, Apache-2.0)
author
{name: string, email?: string}
created
ISO 8601 date (YYYY-MM-DD)
updated
ISO 8601 date (YYYY-MM-DD)
tags
Universal taxonomy: lifecycle, quality-assurance, security, autonomous, multi-agent, evaluation, error-recovery
interface
{input, output, modes[]}
📑
extends (Implementation-Specific)
optional
extends.evaluation
Metrics and thresholds
extends.security
Security standard and scan config
extends.evolution
Evolution triggers and logic
# Universal + Implementation Extension Example extends: evaluation: metrics: [f1, mrr] thresholds: {f1: 0.90, mrr: 0.85} security: standard: CWE scan-on-delivery: true
📝
Example: Code Review Skill
name: code-review-skill version: "1.0.0" description: "Performs comprehensive code review with security focus" # Skill sections sections: introduction: description: "Automated code review assistant" content: | I review code for bugs, security issues, and best practices. I provide actionable feedback with severity ratings. usage: description: "How to invoke this skill" content: | Attach code files and say "review this code" or "analyze for security issues" # Trigger patterns triggers: patterns: - "review.*code" - "analyze.*security" - "check.*bugs" # Evaluation criteria evaluation: criteria: - name: "finding_accuracy" weight: 0.4 - name: "clarity" weight: 0.3 - name: "coverage" weight: 0.3 threshold: 750
✅
Validation Rules
Required Fields
name, version, and description are mandatory.
Version Format
Must be valid semantic version (e.g., "1.0.0", "2.1.3").
Evaluation Weights
All criteria weights must sum to 1.0 (tolerance: 0.01).
Section Names
Must be lowercase with hyphens (kebab-case).
Trigger Patterns
Must be valid regex or plain strings.

See documentation for full framework details.
